知识点分布

第一章：

大数据的由来（为什么会产生大数据存储系统？）

大数据存储系统的产生主要是为了突破传统关系型数据库（RDBMS）的局限性，以适应新时代数据的“3V”特性（量大、多样、快速）。

打破“One Size Fits All”的僵局：传统的抽象认为关系型数据库可以解决所有问题，但随着互联网的发展，这种“一刀切”的模式阻碍了技术发展，必须根据不同的业务场景选择不同的存储方案 。

纵向扩展（Vertical Scaling）的瓶颈：

摩尔定律失效：单纯依靠增加单机硬件性能（如“蜘蛛侠”式的个人能力）已无法满足海量数据的处理需求 。
成本与物理限制：单机性能提升存在物理极限，且成本极其昂贵。

横向扩展（Horizontal Scaling）的需求：

需要像“蚂蚁搬家”一样，通过大量普通节点的协作来支撑更大的请求量 。
传统数据库在面对海量数据时，通过“分库分表”等手段虽然能一定程度上解决问题，但会带来数据同步延迟、跨库事务复杂、扩容困难等痛点 。

数据特性的变化：

数据量爆炸式增长（Volume）。
数据类型繁多（Variety），包含大量非结构化数据（如视频、日志），结构化数据仅占10%。
处理速度要求快（Velocity）。

大数据需要怎样的存储系统

统一资源管理：能够对集群内的计算机及存储资源进行统一的调度和监控。
分散存储与统一管理：数据不是集中在一点，而是分散存储在集群内，但逻辑上进行统一管理。
协同工作与负载均衡：集群内的计算机分工协作，共同完成任务，避免单点过载 。
高可用与容错性（分区容错）：当集群中某一台计算机发生故障时，系统仍能保证功能有效，且数据不会丢失。
高伸缩性（Scalability）：可以用简单的方式部署、扩展集群，以及替换故障节点。

NewSQL数据库的特点

NewSQL 数据库的出现是为了解决 NoSQL 数据库在商业应用中的不足，它结合了传统关系型数据库的强约束和分布式系统的扩展性。

产生的背景（对 NoSQL 的反思）：图灵奖得主 Stonebraker 指出 NoSQL 存在三大“罪状”，这也是 NewSQL 致力于解决的问题 ：
缺乏 ACID 支持（No ACID Equals No Interest）：企业级应用需要事务一致性。
低级查询语言（A Low-Level Query Language is Death）：SQL 依然是最高效的标准语言。
缺乏标准（NoSQL Means No Standards）：各家接口不统一增加了使用成本。
核心特点：
强约束（Strong Constraint）：与 NoSQL 的“弱约束”相对，NewSQL 重新支持了 ACID 事务特性。
SQL 支持：保留了标准 SQL 查询语言的便利性。
可扩展性：依然保持了分布式架构的横向扩展能力（这是区别于传统 RDBMS 的关键）。

第二章：

客户/服务器的体系结构（不同体系结构中AP和DP的功能的变化）

应用处理器(AP) 用于完成数据处理的软件，如：处理访问多个场地的请求，查询全局字典中分布信息等。

数据处理器(DP) 负责进行数据管理的软件，类似于一个集中式数据库管理系统（DBMS）。


分库分表架构、存算分离架构的概念及优缺点

无共享存储型（Shared-Nothing）/ 分库分表架构：
概念：各个节点完全对等，互不共享存储。每个节点都有自己的 SQL 引擎、存储引擎和事务引擎。
代表产品：OceanBase。
优点：高可扩展性（水平扩展）、高可用性、高性能、低成本（可运行在普通PC服务器上）。

共享存储型（Shared-Storage）/ 存算分离架构：
概念：底层存储由分布式存储代替，计算层节点共享同一份数据，拥有访问完整数据集的能力 。通常采用“计算-内存-存储分离”模式 。
代表产品：阿里云 PolarDB、Amazon Aurora、Oracle RAC 。
优点：提升了系统的整体可用性和扩展性 ；Oracle RAC 通过共享缓存和分布式锁提供了写能力的扩展 。
缺点：并发事务处理能力受单个节点性能制约（通常区分读写节点和只读节点）。

分布式数据库系统的模式结构

全局外模式 (GES)：全局用户视图，最高层抽象。
全局概念模式 (GCS)：数据库的整体抽象，包含全部数据特性和逻辑结构。
分片模式 (Fragmentation Schema)：DDBS特有。描述全局数据的逻辑划分（如将一个关系分为几个子关系/片段）。
分配模式 (Allocation Schema)：DDBS特有。描述分片后的局部物理分配视图（即片段存放在哪个物理场地）。
局部概念模式 (LCS)：全局概念模式的子集，描述局部场地上的逻辑结构。
局部内模式 (LIS)：定义局部的物理视图。

分布式数据库系统的数据透明性（定义，举例）

数据透明性是指用户在操作数据时，无需了解数据在分布式系统中的具体分布细节。

分片透明性 (Fragmentation Transparency)：
定义：用户不必考虑数据属于哪个分片（子关系）。位于全局概念模式和分片模式之间 。
举例：用户查询 SUPPLIER 表时，不需要知道该表被切分成了 S1 和 S2 两个片段，直接写 Select … From SUPPLIER 。

分配透明性 (Allocation Transparency)：
定义：用户不必考虑数据片段具体存储在哪个物理场地。位于分片模式和分配模式之间 。
举例：用户查询 SUPPLIER1 分片时，不需要知道它存储在“站点1”，系统会自动定位 。

局部映射透明性 (Local Mapping Transparency)：
定义：用户不必考虑数据的局部存储形式。位于分配模式与局部概念模式之间 。
举例：用户需要指定站点（如 SUPPLIER1 AT SITE1），但不需要了解该站点内部的存储细节 。

多数据库集成的方法，区别与联系

概念：在已存在的局部数据库系统（LDBs）之上，为用户提供一个统一的存取环境。它是自下而上 (Bottom-up) 的集成过程 。
与分布式数据库系统 (DDBS) 的区别 ：
设计方式：DDBS 是自上而下 (Top-down) 设计；MDBS 是自下而上集成。
模式定义：DDBS 的全局模式是预先定义的；MDBS 的模式是集成出来的。
同构性：DDBS 通常同构；MDBS 常用于异构数据库集成（通过中间件） 。
用户：MDBS 一定存在局部用户直接访问局部数据库的情况。

第四章：

HBase解决了HDFS的哪些问题?具有哪些特点?

如果仅用HDFS进行数据管理，存在一些问题:
(1)HDFS不支持对数据的随机改写
(2)HDFS没有数据表的概念
(3)HDFS无法针对行数统计、过滤扫描等常见数据查询功能实现快捷操作，一般需要通过Mapreduce实现。


HBase底层采用HDFS存储，但是文件结构和元数据等自身维护。具体来说具有以下特点:
(1)采用面向列加键值对的存储模式
(2)可实现便捷的向扩展
(3)可以实现自动的数据分片
(4)实现较为严格的读写一致性和自动故障转移
HDFS的优势:(大文件存储、多副本、自动分块)

HBase数据库中region的含义及其特性。同一个表不同行的数据可以存放在不同的服务器，同一个表相同行的数据必须存放在相同的服务器。这句话如何理解?

Region是一段数据的集合，表的一部分数据。HBase是一个会自动分片的数据库。一个Region就相当于关系数据库中分区表的一个分区，或者一个分片。

特性：
Region不能跨服务器，一个RegionServer上有一个或多个Region
当数据量增大时，Region会发生分裂
出于负载均衡的需要，Region会发生迁移
Region所有的数据存取操作都是调用HDFS的客户端接口实现的

HBase的读写流程

HBase数据库设计rowkey和列族的原则和方法

RowKey设计原则:
长度原则： rowkey的长度应该尽可能短，因为rowkey会被存储在HBase的内存中，过长的rowkey会占用更多的内存空间。一般不超过16字节
散列原则： rowkey的哈希值应该是随机分布的，这样可以避免数据倾斜问题。不要将时间放在二进制码前面
唯一原则： rowkey必须是唯一的，不能重复。

加盐salting：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了加盐salting的机制。
具体来说，就是在rowkey中添加一个随机数作为前缀，这样可以确保rowkey的哈希值是随机分布的。

预分区：
为了避免数据倾斜问题，HBase引入了预分区的机制。
具体来说，就是在创建表时，就指定好分区的数量，这样可以确保数据在不同的Region之间是均匀分布的。

散列：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了散列的机制。
具体来说，就是在rowkey中添加一个随机数作为后缀，这样可以确保rowkey的哈希值是随机分布的。

反转rowkey：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了反转rowkey的机制。
具体来说，就是在rowkey中添加一个随机数作为后缀，这样可以确保rowkey的哈希值是随机分布的。

第五章：

(1)跳跃表

内存结构

每条链表都是有序的
上层链表元素集合是下层的子集
跳表高度定义为水平链表层数

每层都是有序链表（默认升序）
每个节点包含两个指针

解决的问题类型(快速写入、更新代价低、支持区间查询)

查找和插入的流程(实现原理)

查找 以左上角元素为起点，若当前节点的后继节点值小于等于待查询值，则向右移动，否则向下移动。直到查询到结果为止。
插入 查找到待插入位置后，随机决定新节点的层数（通过算法计算，通常是1/2或1/4的概率提升该节点的层数），并在相应层数分别插入新节点。

(2)LSMI树

内存+外存结构

内存就是跳表，这个跳表的key是HBase中key-value的key的构成，包括rowkey，family，Qualifier，Timestamp，Type。

外存就是磁盘，内存满了之后，会将内存中的数据写入磁盘。
为了避免flush操作影响性能，会把当前内存设为snapshot，不允许在这一个内存中写入数据，新开辟一块内存来写入数据，当snapshot写入完毕之后对应的内存即可释放。
整个写入过程，LSM树都是append过程，顺序写入，没有随机读写。

什么是compaction?分为哪两种类型?各自优缺点?

compaction将key相同的数据综合起来
分为minor compaction和major compaction。

minor compaction：
优点：局部的compact操作，少了IO减少文件个数，提升读取性能
缺点：全局操作无法在合并过程中完成

major compaction：
优点：合并之后只有一个文件，读取性能最高。
缺点：合并所有的文件需要很长时间，消耗大量带宽。

为什么说LSM树是一种写入友好的数据结构?

因为是顺序写入，提高了写入的效率，但是对读取操作不友好。
异步compaction提高了读取性能。

(3)布隆过滤器

解决的问题类型

高效判断元素是否在集合中

构造方法和查询过程(实现原理)

搞一个长度为N的bit数组，全部初始化为0。
对于一个元素，将其进行k次哈希，将哈希值对N取模，将得到的k个位置的bit数组设为1。
查询时，对元素进行k次哈希，将哈希值对N取模，若得到的k个位置的bit数组全为1，则认为元素可能在集合中，否则一定不在集合中。

不同索引结构的适用场景


索引结构 核心特点 适用场景 典型应用
哈希索引 (Hash) $O(1)$ 极速查找，不支持范围查询 只有等值查询 (Key=Value) Redis, HashMap
B+ 树 读写平衡，层高固定，适合读多写少 传统关系型数据库，强一致性，稳定读 MySQL, Oracle
LSM 树 写性能极强，读性能稍弱（需合并） 海量数据写入，日志存储，历史数据归档 HBase, Cassandra, RocksDB
跳跃表 实现简单，支持并发和区间查询 内存中的有序数据维护 HBase MemStore, Redis ZSet


第六章：

(1)嵌套事务的概念

嵌套事务是指在一个事务内部又包含一个或多个子事务的事务模型。
子事务可以是独立的事务，也可以是嵌套在父事务中的事务。

(2)分布式数据库的一致性级别的内容，并可举例说明

强一致性 (Strong Consistency / Linearizability)
定义：写操作完成后，系统保证后续所有的读操作都能读到这个最新的值。这就好像系统只有一份数据副本一样。
举例：银行转账系统、关系型数据库（RDBMS）的主从同步（同步模式）。

弱一致性 (Weak Consistency)
定义：写操作完成后，系统不能保证后续的读操作能立即读到最新值，也不承诺多久能读到。
举例：缓存系统（如 Memcached），为了极高的性能，允许短时间内读到旧数据。

最终一致性 (Eventual Consistency) —— 弱一致性的特例，最常见
定义：系统保证在没有新的更新操作下，经过一段时间（不一致窗口），所有副本最终会达到一致的状态。
举例：DNS系统（修改域名解析可能需要几分钟生效）、CDN（内容分发）、微信朋友圈点赞（你点了赞，朋友可能过几秒才看到）。

(3)分布式数据库的CAP理论和BASE理论

CAP理论：
定义：在一个分布式系统中，最多只能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）这三个需求中的两个。

  • 一致性：所有节点在同一时间看到的数据是一致的。
  • 可用性：每个请求都能收到响应，无论是否成功。
  • 分区容错性：系统在网络分区的情况下仍能正常工作。

HBase是一个CP系统，牺牲了部分可用性保证系统的一致性。
HBase中的每一条数据只会出现在一个Region中，数据冗余备份不是在Region这个层面做的，而是依赖HDFS。

BASE理论：
定义：BASE是对CAP理论的一种补充，强调在分布式系统中，为了满足可用性和分区容错性，而放弃一致性。即使无法做到强一致，但每个应用都可以通过自身业务特点，采用适当的方式来使系统达到最终一致性。

  • 基本可用（Basic Availability）：系统在出现故障时，仍然能正常运行，只是响应时间会延长。允许损失部分可用性但绝不等同于不可用。
  • 软状态（Soft State）：允许存在中间状态，且该中间状态不影响系统的整体可用性。
  • 最终一致性（Eventual Consistency）：系统最终会达到一致状态，只是需要一定的时间。

(4)分布式事务提交协议(两阶段提交协议执行流程，存在的问题，解决的方法)

协调者：负责协调分布式事务的提交和回滚。
参与者：负责执行分布式事务的操作，向协调者报告执行结果。
协调者和参与者都有本地日志，他们在进行操作前都必须将该操作记录到相应的日志文件中，以进行事务故障恢复和系统故障恢复。

执行流程

协调者在征求各参与者的意见之前，首先要在它的日志文件中写入一条“开始提交”（Begin_commit）记录。然后，协调者向所有参与者发送“预提交”（Prepare）命令。协调者进入等待状态，等待收集各参与者的应答。

参与者接收到“预提交”（Prepare）命令后，判断本场地子事务是否已准备好提交。
若是，参与者在日志中写入“准备提交”（Ready）记录，并向协调者发送“准备提
交”（Ready）应答；
否则，参与者在日志中写入“准备废弃”（Abort）记录，并向协调者发送“准备废弃”（Abort）应答。发送应答后，参与者将等待协调者所做出的最终决定。

协调者收集各参与者发来的应答，若存在至少一个参与者发来“准备废弃”的应答，则协调者在日志中写入“决定废弃”（Abort）记录，并发送“全局废弃”（Abort）命令；
否则，协调者在日志中写入“决定提交”（Commit）记录，向所有参与者发送“全局提交”（Commit）命令。此后，协调者进入等待状态，等待收集各参与者的应答信息。

参与者接收到协调者发来的命令，若为“全局提交”命令，则参与者在日志中写入“提交”（Commit）的记录，并对子事务实施提交；
否则，参与者在日志文件中写入“废弃”（Abort）的记录，并对子事务实施废弃。实施完毕后，各个参与者要向协调者发送确认信息（Ack）。

当协调者接收到所有参与者发送的确认信息后，在日志中写入“事务结束”（End_transaction）记录，全局事务结束。

存在的问题：

  1. 同步阻塞
  2. 单点问题
  3. 过于保守，实现的CP系统

如果在两段提交协议执行的过程中出现协调者故障或网络故障，那么参与者就不能及时收到协调者发出的“提交/废弃(C/A)”命令，参与者处的子事务将处于等待状态。
在故障恢复前，参与者子事务所占有的系统资源也不能被释放，参与者的子事务进入了阻塞状态。
若参与者一直收不到协调者的命令，则子事务将始终处于阻塞状态而挂在相应的执行场地上，所占用的系统资源也不能被其它事务利用。

两阶段提交协议的终结协议的核心思想是：参与者通过访问其它参与者的当前状态来推断协调者的决定，从而确定终结类型（全局提交还是全局废弃）。

关于步骤三，因为参与者是提交状态必然是协调者收到了所有参与者的准备就绪命令才能发出提交命令，进而使参与者变为提交状态，所以新的协调者可以发送commit命令。

(5)HBase的一致性ACID特性的实现方法

原子性 (Atomicity)：

  • 实现：通过 WAL (Write-Ahead Log) 和 RowLock (行锁)。
  • 写入前先拿行锁，然后写 WAL。只要 WAL 写入成功，即使内存数据没刷盘机器断电，重启后也能通过日志重放恢复，保证“要么全成功，要么全失败”。

一致性 (Consistency)：

  • 实现：HBase 是强一致性系统。通过 MVCC (多版本并发控制) 机制。
  • 读操作只能读到 ReadPoint（当前已完成事务的最大版本号）之前的数据，未完成的中间状态数据不可见。

隔离性 (Isolation)：

  • 实现：通过 MVCC 实现 Read Committed 级别的隔离。
  • 写操作通过加锁互斥；读操作读取旧版本数据，读写不冲突，提高了并发度。

持久性 (Durability)：

  • 实现：通过 HDFS 多副本机制 + WAL。数据最终落在磁盘的 HFile 上，且在 HDFS 有 3 副本冗余。

(6)三阶段提交协议与两阶段提交协议的区别

三个阶段：

  1. 投票表决阶段 (Prepare Phase)：协调者向所有参与者发送“预提交”（Prepare）命令，参与者响应是否准备好提交。不锁定资源，不等待其他事务。
  2. 提交阶段 (Commit Phase)：协调者收集所有参与者的“准备就绪”（Ready）应答，若存在至少一个参与者发来“准备废弃”的应答，则协调者在日志中写入“决定废弃”（Abort）记录，并发送“全局废弃”（Abort）命令；否则，协调者在日志中写入“决定提交”（Commit）记录，向所有参与者发送“全局提交”（Commit）命令。
  3. 确认阶段 (Confirm Phase)：参与者接收到协调者发来的命令，若为“全局提交”命令，则参与者在日志中写入“提交”（Commit）的记录，并对子事务实施提交；否则，参与者在日志文件中写入“废弃”（Abort）的记录，并对子事务实施废弃。实施完毕后，各个参与者要向协调者发送确认信息（Ack）。

三阶段提交协议（3PC）是二阶段提交协议（2PC）的一个改进版本，它通过增加一个预提交阶段和引入参与者超时机制，有效地解决了2PC中因协调者单点故障而导致的整个系统无限期阻塞这一核心可用性问题。然而，它是以增加协议复杂度、通信开销和可能引入网络分区下不一致为代价的。

多了“预提交”阶段 (PreCommit)：
2PC：Prepare $\rightarrow$ Commit
3PC：CanCommit $\rightarrow$ PreCommit $\rightarrow$ DoCommit

引入了超时机制：在 2PC 中，只有协调者有超时机制。
在 3PC 中，参与者也有超时机制。如果参与者在 PreCommit 阶段之后收不到协调者的死活消息，它会默认执行提交（这就避免了资源长期被锁死）。
非阻塞：由于多了一个阶段做缓冲，且有超时提交策略，3PC 在协调者故障时，参与者能达成一致避免长期阻塞。
