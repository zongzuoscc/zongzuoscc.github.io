知识点分布

第一章：

大数据的由来（为什么会产生大数据存储系统？）

大数据存储系统的产生主要是为了突破传统关系型数据库（RDBMS）的局限性，以适应新时代数据的“3V”特性（量大、多样、快速）。

打破“One Size Fits All”的僵局：传统的抽象认为关系型数据库可以解决所有问题，但随着互联网的发展，这种“一刀切”的模式阻碍了技术发展，必须根据不同的业务场景选择不同的存储方案 。

纵向扩展（Vertical Scaling）的瓶颈：

摩尔定律失效：单纯依靠增加单机硬件性能（如“蜘蛛侠”式的个人能力）已无法满足海量数据的处理需求 。
成本与物理限制：单机性能提升存在物理极限，且成本极其昂贵。

横向扩展（Horizontal Scaling）的需求：

需要像“蚂蚁搬家”一样，通过大量普通节点的协作来支撑更大的请求量 。
传统数据库在面对海量数据时，通过“分库分表”等手段虽然能一定程度上解决问题，但会带来数据同步延迟、跨库事务复杂、扩容困难等痛点 。

数据特性的变化：

数据量爆炸式增长（Volume）。
数据类型繁多（Variety），包含大量非结构化数据（如视频、日志），结构化数据仅占10%。
处理速度要求快（Velocity）。

大数据需要怎样的存储系统
统一资源管理：能够对集群内的计算机及存储资源进行统一的调度和监控。
分散存储与统一管理：数据不是集中在一点，而是分散存储在集群内，但逻辑上进行统一管理。
协同工作与负载均衡：集群内的计算机分工协作，共同完成任务，避免单点过载 。
高可用与容错性（分区容错）：当集群中某一台计算机发生故障时，系统仍能保证功能有效，且数据不会丢失。
高伸缩性（Scalability）：可以用简单的方式部署、扩展集群，以及替换故障节点。

NewSQL数据库的特点

NewSQL 数据库的出现是为了解决 NoSQL 数据库在商业应用中的不足，它结合了传统关系型数据库的强约束和分布式系统的扩展性。

产生的背景（对 NoSQL 的反思）：图灵奖得主 Stonebraker 指出 NoSQL 存在三大“罪状”，这也是 NewSQL 致力于解决的问题 ：
缺乏 ACID 支持（No ACID Equals No Interest）：企业级应用需要事务一致性。
低级查询语言（A Low-Level Query Language is Death）：SQL 依然是最高效的标准语言。
缺乏标准（NoSQL Means No Standards）：各家接口不统一增加了使用成本。
核心特点：
强约束（Strong Constraint）：与 NoSQL 的“弱约束”相对，NewSQL 重新支持了 ACID 事务特性。
SQL 支持：保留了标准 SQL 查询语言的便利性。
可扩展性：依然保持了分布式架构的横向扩展能力（这是区别于传统 RDBMS 的关键）。

第二章：

客户/服务器的体系结构（不同体系结构中AP和DP的功能的变化）

应用处理器(AP) 用于完成数据处理的软件，如：处理访问多个场地的请求，查询全局字典中分布信息等。

数据处理器(DP) 负责进行数据管理的软件，类似于一个集中式数据库管理系统（DBMS）。


分库分表架构、存算分离架构的概念及优缺点

无共享存储型（Shared-Nothing）/ 分库分表架构：
概念：各个节点完全对等，互不共享存储。每个节点都有自己的 SQL 引擎、存储引擎和事务引擎。
代表产品：OceanBase。
优点：高可扩展性（水平扩展）、高可用性、高性能、低成本（可运行在普通PC服务器上）。

共享存储型（Shared-Storage）/ 存算分离架构：
概念：底层存储由分布式存储代替，计算层节点共享同一份数据，拥有访问完整数据集的能力 。通常采用“计算-内存-存储分离”模式 。
代表产品：阿里云 PolarDB、Amazon Aurora、Oracle RAC 。
优点：提升了系统的整体可用性和扩展性 ；Oracle RAC 通过共享缓存和分布式锁提供了写能力的扩展 。
缺点：并发事务处理能力受单个节点性能制约（通常区分读写节点和只读节点）。

分布式数据库系统的模式结构

全局外模式 (GES)：全局用户视图，最高层抽象。
全局概念模式 (GCS)：数据库的整体抽象，包含全部数据特性和逻辑结构。
分片模式 (Fragmentation Schema)：DDBS特有。描述全局数据的逻辑划分（如将一个关系分为几个子关系/片段）。
分配模式 (Allocation Schema)：DDBS特有。描述分片后的局部物理分配视图（即片段存放在哪个物理场地）。
局部概念模式 (LCS)：全局概念模式的子集，描述局部场地上的逻辑结构。
局部内模式 (LIS)：定义局部的物理视图。

分布式数据库系统的数据透明性（定义，举例）
数据透明性是指用户在操作数据时，无需了解数据在分布式系统中的具体分布细节。
分片透明性 (Fragmentation Transparency)：
定义：用户不必考虑数据属于哪个分片（子关系）。位于全局概念模式和分片模式之间 。
举例：用户查询 SUPPLIER 表时，不需要知道该表被切分成了 S1 和 S2 两个片段，直接写 Select … From SUPPLIER 。
分配透明性 (Allocation Transparency)：
定义：用户不必考虑数据片段具体存储在哪个物理场地。位于分片模式和分配模式之间 。
举例：用户查询 SUPPLIER1 分片时，不需要知道它存储在“站点1”，系统会自动定位 。
局部映射透明性 (Local Mapping Transparency)：
定义：用户不必考虑数据的局部存储形式。位于分配模式与局部概念模式之间 。
举例：用户需要指定站点（如 SUPPLIER1 AT SITE1），但不需要了解该站点内部的存储细节 。

多数据库集成的方法，区别与联系

概念：在已存在的局部数据库系统（LDBs）之上，为用户提供一个统一的存取环境。它是自下而上 (Bottom-up) 的集成过程 。
与分布式数据库系统 (DDBS) 的区别 ：
设计方式：DDBS 是自上而下 (Top-down) 设计；MDBS 是自下而上集成。
模式定义：DDBS 的全局模式是预先定义的；MDBS 的模式是集成出来的。
同构性：DDBS 通常同构；MDBS 常用于异构数据库集成（通过中间件） 。
用户：MDBS 一定存在局部用户直接访问局部数据库的情况。

第四章：

HBase解决了HDFS的哪些问题?具有哪些特点?

如果仅用HDFS进行数据管理，存在一些问题:
(1)HDFS不支持对数据的随机改写
(2)HDFS没有数据表的概念
(3)HDFS无法针对行数统计、过滤扫描等常见数据查询功能实现快捷操作，一般需要通过Mapreduce实现。


HBase底层采用HDFS存储，但是文件结构和元数据等自身维护。具体来说具有以下特点:
(1)采用面向列加键值对的存储模式
(2)可实现便捷的向扩展
(3)可以实现自动的数据分片
(4)实现较为严格的读写一致性和自动故障转移
HDFS的优势:(大文件存储、多副本、自动分块)

HBase数据库中region的含义及其特性。同一个表不同行的数据可以存放在不同的服务器，同一个表相同行的数据必须存放在相同的服务器。这句话如何理解?
Region是一段数据的集合，表的一部分数据。HBase是一个会自动分片的数据库。一个Region就相当于关系数据库中分区表的一个分区，或者一个分片。

特性：
Region不能跨服务器，一个RegionServer上有一个或多个Region
当数据量增大时，Region会发生分裂
出于负载均衡的需要，Region会发生迁移
Region所有的数据存取操作都是调用HDFS的客户端接口实现的

HBase的读写流程

HBase数据库设计rowkey和列族的原则和方法
RowKey设计原则:
长度原则：rowkey的长度应该尽可能短，因为rowkey会被存储在HBase的内存中，过长的rowkey会占用更多的内存空间。一般不超过16字节
散列原则：rowkey的哈希值应该是随机分布的，这样可以避免数据倾斜问题。不要将时间放在二进制码前面
唯一原则：rowkey必须是唯一的，不能重复。

加盐salting：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了加盐salting的机制。
具体来说，就是在rowkey中添加一个随机数作为前缀，这样可以确保rowkey的哈希值是随机分布的。

预分区：
为了避免数据倾斜问题，HBase引入了预分区的机制。
具体来说，就是在创建表时，就指定好分区的数量，这样可以确保数据在不同的Region之间是均匀分布的。

散列：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了散列的机制。
具体来说，就是在rowkey中添加一个随机数作为后缀，这样可以确保rowkey的哈希值是随机分布的。

反转rowkey：
为了避免rowkey的哈希值分布不均匀，导致数据倾斜问题，HBase引入了反转rowkey的机制。
具体来说，就是在rowkey中添加一个随机数作为后缀，这样可以确保rowkey的哈希值是随机分布的。
